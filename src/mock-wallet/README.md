# Mock EUDI Wallet

Simulates an EU Digital Identity Wallet for the PoC demo. Issues SD-JWT-VC credentials locally (simulating carrier issuance), derives per-service pseudonyms, and presents credentials via OpenID4VP to the provisioning agent. Runs on the host — not in Docker — because it needs terminal access for the consent TUI.

## Protocols implemented

**SD-JWT-VC (RFC 9901)** — credential format:
- Issuer-signed JWT with `typ: vc+sd-jwt`, ES256 signing, `cnf` claim binding holder public key
- Selectively-disclosable claims via `_sd` array: `msisdn` (phone number) and `pseudonym`
- Disclosure format: base64url-encoded `[salt, claim_name, claim_value]` arrays, SHA-256 hashed in `_sd`
- Key Binding JWT (`typ: kb+jwt`) appended at presentation, signed by holder key, containing `nonce`, `aud`, `iat`, `sd_hash`
- Credential type: `urn:eu.europa.ec.eudi:msisdn:1` (matches EU reference wallet MSISDN credential)

**OpenID4VP** — credential presentation:
- RP-initiated, same-device flow only. Browser redirect to `/authorize` with `request_uri` + `state`
- Wallet fetches authorization request object from `request_uri` (contains `nonce`, `response_uri`, `client_id`, `presentation_definition`)
- Wallet builds VP token (SD-JWT-VC + KB-JWT) and POSTs to `response_uri`
- Verifier returns `redirect_uri`; wallet redirects browser there

**Not implemented:** cross-device flow, holder-initiated presentation, credential offer/issuance protocol (OID4VCI), status lists, trust chains, batch presentation, multiple credentials per request.

## What is mocked

Carrier credential issuance happens locally — `carrier.py` loads a test carrier key and mints SD-JWT-VCs directly. A real deployment would use OID4VCI from a carrier credential issuer. The carrier key, holder keys, and their trust relationship are all local test fixtures generated by the stack's key setup script.

## Why this subset

The demo proves one path: credential issuance -> selective disclosure -> presentation -> verification -> Matrix account provisioning -> federated messaging. This is the minimum OpenID4VP surface that exercises the full chain. Everything omitted (cross-device, holder-initiated, trust frameworks) is orthogonal to the registration flow being demonstrated.

## Pseudonym derivation

Wallet-side, deterministic, per-service:

```
pseudonym = SHA256(SHA256(holder_private_key_pem) || client_id)
```

The holder's private key is hashed first (domain separation), then concatenated with the verifier's `client_id` and hashed again. Same wallet presenting to the same service always produces the same pseudonym. Different services get different pseudonyms. The pseudonym is included as a selectively-disclosable claim in the SD-JWT-VC — the provisioning agent receives it after verification, never derives it.

This derivation is a PoC stand-in. The provisioning agent trusts the wallet's pseudonym assertion without cryptographic proof of correct derivation. In production, SGVP zero-knowledge proofs or BBS+ per-verifier linkability would bind the pseudonym to the wallet's credential-committed secret. The mechanism is under active specification in [ARF Discussion #375](https://github.com/eu-digital-identity-wallet/eudi-doc-architecture-and-reference-framework/discussions/375).

## Running

```
cd src/mock-wallet
uv run uvicorn wallet:app --host 0.0.0.0 --port 8095
```

Or use `services/scripts/start.sh` which starts the full stack including this wallet. The round-trip test runs standalone:

```
cd src/mock-wallet
uv run python test_roundtrip.py
```
